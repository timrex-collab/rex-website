# Pre-Push-Verification (PFLICHT — D16-Lesson Learned)

**Erst nach erfolgreicher Verifikation oben durchführen. Niemals „Sync Changes" in Replit verwenden.**

```bash
# Schritt 1 — gezieltes git add der 7 erwarteten Dateien
git add client/public/images/dach-photovoltaik-bochum-hero-braas-pv-premium-turmalin.webp \
        client/public/images/braas-pv-premium-turmalin-dachintegrierte-photovoltaik-bochum.webp \
        client/public/images/braas-pv-premium-tegalit-dachintegrierte-photovoltaik-bochum.webp \
        client/public/images/konventionelle-aufdach-pv-photovoltaikanlage-dach-bochum.webp \
        client/src/pages/DachPhotovoltaikBochum.tsx \
        client/src/components/Navigation.tsx \
        client/src/pages/Services.tsx

# Schritt 2 — Vorhandene Commits prüfen (Auto-Checkpoint-Schutz)
git log --oneline origin/main..HEAD
# Erwartung: leer.
# Falls Output: STOP — ungewollter Commit liegt vor → git reset --soft origin/main, dann von vorne.

# Schritt 3 — Commit
git commit -m "feat: D24 — PV-Bilder aktualisieren, VELUX Preisrechner in Navigation und Gründach in Leistungen"

# Schritt 4 — Inhalt des Commits prüfen
git show --stat HEAD
# Erwartung: exakt 7 Dateien (4 WebP + 3 TSX).
# Falls mehr/weniger: STOP, prüfen und ggf. git reset --soft origin/main.
# Falls attached_assets/Pasted-*.txt enthalten: git reset --soft origin/main → git restore --staged attached_assets/Pasted-*.txt → recommit.

# Schritt 5 — Push
git push
```

# Git-Hygiene bei Auto-Checkpoint (Fallback)

Falls Replit einen Auto-Checkpoint erzeugt oder `attached_assets` ins Staging geraten ist, bevor der Pre-Push-Block ausgeführt wird:

```bash
git reset --soft origin/main
git restore --staged attached_assets 2>/dev/null || true
git restore attached_assets 2>/dev/null || true
```

Danach **Pre-Push-Verification (Schritte 1–5) von vorne** ausführen.
